# Plan 5.1: API Routes for Vaults and Agents

## Objective
Create the thin API route handlers that the dashboard will call to fetch vault data and trigger agent runs.

## Context
- src/lib/db/vault-service.ts (all DB access)
- src/lib/db/client.ts (Prisma singleton)
- src/lib/queue/client.ts (BullMQ queues)
- docs/architecture.md §6 (API Layer Route Map)

## Tasks

<task type="auto">
  <name>Create GET /api/vaults route</name>
  <files>src/app/api/vaults/route.ts</files>
  <action>
    Returns all clients with compliance/onboarding summary.
    - Query all clients via Prisma (direct, not VaultService — cross-client list)
    - For each client, include: id, name, email, accountType, onboardingStatus, onboardingStage, advisor.name, firm.name, _count of documents, urgencyCount derived from document statuses (EXPIRED/EXPIRING_SOON/MISSING)
    - Return `NextResponse.json({ clients })` with 200
    - No auth yet — plain data endpoint
  </action>
  <verify>npx tsc --noEmit</verify>
  <done>Route compiles and returns typed response.</done>
</task>

<task type="auto">
  <name>Create GET /api/vaults/[clientId] route</name>
  <files>src/app/api/vaults/[clientId]/route.ts</files>
  <action>
    Full vault detail for one client.
    - Construct `VaultService({ clientId })` and call `getClientProfile()`, `getDocuments()`, `getActionHistory()`
    - Return `{ client, documents, recentActions }` (actions limited to 20)
    - Return 404 with JSON error if clientId not found
  </action>
  <verify>npx tsc --noEmit</verify>
  <done>Route returns full vault detail with error handling.</done>
</task>

<task type="auto">
  <name>Create POST /api/vaults/[clientId]/upload route</name>
  <files>src/app/api/vaults/[clientId]/upload/route.ts</files>
  <action>
    Mock document upload trigger.
    - Parse body: `{ documentType, fileRef? }`
    - Upsert document with `PENDING_REVIEW` status via VaultService
    - Enqueue priority job on `priority-queue` for this clientId (agentType COMPLIANCE or ONBOARDING based on document type)
    - Return 202 `{ enqueued: true, jobId }`
  </action>
  <verify>npx tsc --noEmit</verify>
  <done>Upload route upserts document and enqueues BullMQ job.</done>
</task>

<task type="auto">
  <name>Create GET /api/agents/status route</name>
  <files>src/app/api/agents/status/route.ts</files>
  <action>
    Returns current queue depth and recent job counts.
    - Get counts from `priority-queue` and `scheduled-queue` BullMQ queues: waiting, active, completed, failed
    - Return `{ priorityQueue, scheduledQueue }` with counts
  </action>
  <verify>npx tsc --noEmit</verify>
  <done>Status route returns queue depths.</done>
</task>

<task type="auto">
  <name>Create POST /api/agents/trigger route</name>
  <files>src/app/api/agents/trigger/route.ts</files>
  <action>
    Manual agent trigger from dashboard.
    - Parse body: `{ clientId, agentType: "COMPLIANCE" | "ONBOARDING" }`
    - Validate with Zod
    - Enqueue to `priority-queue` with trigger type `MANUAL`
    - Return 202 `{ enqueued: true, jobId }`
  </action>
  <verify>npx tsc --noEmit</verify>
  <done>Trigger route validates input and enqueues job.</done>
</task>

## Success Criteria
- [ ] All 5 API routes compile cleanly.
- [ ] Vault list route returns all clients with urgency counts.
- [ ] Vault detail uses VaultService (never raw Prisma).
- [ ] Upload and trigger routes return 202 with jobId.
